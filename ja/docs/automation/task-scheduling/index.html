タスクスケジューリング

aiFetchly のタスクスケジューラーを使えば、指定した時刻にタスクを自動実行したり、他のタスクの完了に応じて実行したりすることで、マーケティングワークフローを自動化できます。定期的な検索、自動メールキャンペーン、複雑なマルチステップワークフローを設定できます。

スケジューリングについて​

タスクスケジューラーは 3 種類のタスクトリガーをサポートしています:

トリガータイプ説明最適な用途
Croncron 式による時間ベースのスケジューリング定期タスク、毎日/毎週/毎月のジョブ
依存関係別のタスクの完了によってトリガーマルチステップワークフロー、タスクチェーン
手動手動でトリガーされた場合のみ実行オンデマンドタスク、テスト
すべてを自動化

リードジェネレーションからメールキャンペーンまで、スケジューリングの自動化は時間を節約し、マーケティングワークフローの安定した実行を確保します。

スケジュールタスクの作成​

ステップ 1: スケジューラーに移動​

  1. 左側のナビゲーションメニューで Schedule をクリック
  2. New Schedule ボタンをクリック

ステップ 2: 基本情報​

以下の必須情報を入力します:

スケジュール名​

  • 目的: リスト内でスケジュールを識別するための名前
  • 例: 「毎日の Google 検索」、「週次メールキャンペーン」
  • 必須: はい

タスクタイプ​

スケジュールするタスクの種類を選択:

  • Search: 検索エンジンスクレイピングタスク
  • Email Extract: メール抽出タスク
  • Bulk Email: メールマーケティングキャンペーン
  • Yellow Pages: ディレクトリスクレイピングタスク
  • Video Download: 動画ダウンロードタスク

タスク ID​

  • 目的: 特定のタスクインスタンスにリンク
  • 選択: 選択したタイプの既存タスクから選択
  • 必須: はい

説明​

  • 目的: スケジュールの目的に関するコンテキストを提供
  • 例: 「ターゲット都市の新しいマーケティング代理店を毎日検索」
  • 任意: はい

ステップ 3: トリガーの設定​

Cron スケジューリング（時間ベース）​

Enable Cron を有効にしてスケジュールを設定:

プリセットオプション:

  • 毎分
  • 毎時
  • 毎日（深夜）
  • 毎週（日曜深夜）
  • 毎月（1 日深夜）
  • 15 分ごと
  • 30 分ごと
  • 2 時間ごと
  • 6 時間ごと
  • 12 時間ごと
  • 平日 9 時
  • 週末 10 時

カスタム Cron ビルダー:

フィールドオプション説明
分*/5、*/15、*/30、または特定の分5/15/30 分ごと、または特定の時刻
時*/2、*/6、*/12、または特定の時2/6/12 時間ごと、または特定の時刻
日*/2 または特定の日2 日ごと、または特定の日
月*/3、*/6、または特定の月3/6 か月ごと、または特定の月
曜日1-5（平日）、0,6（週末）、または特定平日、週末、または特定の曜日

Cron 式の形式:

┌───────────── 分 (0 - 59)
│ ┌───────────── 時 (0 - 23)
│ │ ┌───────────── 日 (1 - 31)
│ │ │ ┌───────────── 月 (1 - 12)
│ │ │ │ ┌───────────── 曜日 (0 - 6) (日曜〜土曜)
│ │ │ │ │
* * * * *

例:

0 9 * * 1-5      # 平日 9:00
*/30 * * * *     # 30 分ごと
0 0 * * 1        # 毎週月曜日の深夜
0 9,15 * * *     # 毎日 9:00 と 15:00
0 0 1 * *        # 毎月 1 日の深夜

次回実行時刻:

  • システムが次回の実行時刻を自動的に計算して表示
  • cron 式の検証に役立ちます

依存関係スケジューリング（タスクベース）​

Enable Dependency を有効にして、別のタスクの完了時にこのタスクをトリガーします。

設定:

  1. Parent Schedule: このタスクをトリガーするスケジュールを選択

  2. 依存条件:

    • On Success: 親タスクが成功した場合のみ実行
    • On Completion: 親タスクの完了後（成功・失敗を問わず）に実行
    • On Failure: 親タスクが失敗した場合のみ実行

  3. 遅延（分）: 親タスク完了後、このタスクを実行するまでの待機時間

    • 0 分: 直ちに実行
    • 5 分: 5 分待機してから開始
    • 60 分: 1 時間待機してから開始

ユースケース:

  • 抽出後のメール送信: メールを抽出し、完了時にキャンペーンを送信
  • スクレイピング後の分析: データをスクレイピングし、AI 分析を実行
  • マルチステージキャンペーン: 初回アウトリーチ → フォローアップ 1 → フォローアップ 2
依存関係チェーン

マルチレベルの依存関係チェーンを作成できます:

  • タスク A（Cron）→ タスク B（依存）→ タスク C（依存）

これにより強力な自動化ワークフローが構築できます。

手動実行​

Enable Manual を有効にして、トリガー時のみタスクを実行:

  • 自動スケジューリングなし
  • "Run Now" ボタンでオンデマンド実行
  • テストや頻度の低いタスクに便利

ステップ 4: アクティブステータス​

スケジュールを以下のように切替:

  • Active: トリガー設定に従ってスケジュールが実行
  • Inactive: スケジュールが無効化され、実行されない

ステップ 5: スケジュールの保存​

Save をクリックしてスケジュールを作成します。以下の操作が可能:

  • 後で 編集
  • 必要に応じて 有効化/無効化
  • Run Now で即座にテスト

スケジュールタスクの管理​

スケジュールリストの表示​

Schedule に移動してすべてのスケジュールタスクを表示します。

スケジュール概要:

  • ステータスカード: 合計、アクティブ、非アクティブ、一時停止の各スケジュール数を表示
  • フィルター: 名前、ステータス、タスクタイプ、トリガータイプで検索

スケジュールテーブル​

スケジュールリストには以下が表示されます:

列説明
Nameタスクタイプインジケーター付きのスケジュール名
StatusActive（緑）、Inactive（グレー）、Paused（黄）
Trigger TypeCron、Dependency、または Manual
ScheduleCron 式または依存関係の説明
Next Run次回実行時刻とカウントダウン
Last Run直近の実行と経過時間
Executions成功数 / 失敗数
Actions編集、削除、一時停止/再開、Run Now

スケジュールアクション​

アクション説明利用可能なタイミング
Editスケジュール設定を変更常時
Deleteスケジュールを削除常時
Enable非アクティブなスケジュールを有効化非アクティブなスケジュール
Disableスケジュールを無効化アクティブなスケジュール
Pause実行を一時的に停止アクティブなスケジュール
Resume一時停止中のスケジュールを再開一時停止中のスケジュール
Run Now直ちに実行常時

実行の監視​

実行履歴の表示​

  1. スケジュールの View Details をクリック
  2. 実行履歴テーブルを表示

実行情報:

列説明
Start Time実行の開始時刻
End Time実行の完了時刻（または "Running..."）
Duration実行にかかった時間（またはライブカウンター）
StatusSuccess（緑）、Failed（赤）、Running（青）
Result実行結果のサマリー
Errorエラーメッセージ（実行が失敗した場合）
Actions詳細表示、実行中のキャンセル

実行統計​

詳細表示には以下が表示されます:

  • 成功率: 成功した実行の割合
  • 平均所要時間: 通常の実行時間
  • 総実行数: 全体の件数と内訳
  • 直近のエラー: 最新の失敗メッセージ
  • 次回実行: 次回実行までのカウントダウン

リアルタイム監視​

実行中のタスクについて:

  • リアルタイム所要時間: 経過時間をカウンターで表示
  • キャンセルボタン: 必要に応じて実行を停止
  • 自動更新: 数秒ごとにステータスが更新

一般的なスケジューリングパターン​

パターン 1: 毎日のリードジェネレーション​

スケジュール: 平日毎日 9:00

Cron: 0 9 * * 1-5
Task: 検索エンジンスクレイピング

ユースケース: 毎朝セールスチームに新鮮なリードを提供。

パターン 2: 毎時の監視​

スケジュール: 営業時間中は毎時

Cron: 0 9-17 * * 1-5
Task: Search / Yellow Pages

ユースケース: 営業時間中に新しいビジネスリストを監視。

パターン 3: 週次キャンペーン​

スケジュール: 毎週月曜日 10:00

Cron: 0 10 * * 1
Task: Bulk Email Sending

ユースケース: 週次メールニュースレターまたはアウトリーチキャンペーン。

パターン 4: マルチステップワークフロー​

タスク A（Cron）: 毎日 9:00 - リードを検索 タスク B（依存）: A の成功後 - メールを抽出 タスク C（依存）: B の成功後（30 分遅延） - メールを送信

ユースケース: 自動化されたリードジェネレーションとアウトリーチパイプライン。

パターン 5: メンテナンスタスク​

スケジュール: 毎週日曜日 3:00

Cron: 0 3 * * 0
Task: データクリーンアップまたはバックアップ

ユースケース: トラフィックの少ない時間帯の定期的なメンテナンス。

ベストプラクティス​

1. スケジュール設計​

重複を回避:

  • 次回のスケジュール実行前にタスクが完了することを確認
  • 頻度の設定時に平均実行時間を考慮
  • 重複するタスクの順序付けに依存関係を使用

オフピーク時間:

  • リソース集約型のタスクは時間外にスケジュール
  • ユーザーアクティビティとの競合を避ける
  • グローバル運用ではタイムゾーンを考慮

バッファ時間:

  • 依存タスク間にバッファを追加
  • 実行時間の変動を考慮
  • 連鎖的な遅延を防止

2. エラー処理​

失敗を監視:

  • 実行履歴を定期的に確認
  • 繰り返される失敗を調査
  • 必要に応じてスケジュールやタスクを調整

アラートの設定:

  • スケジュールを毎週確認
  • 停止または一時停止中のタスクをチェック
  • 依存関係が正しく発火しているか確認

段階的縮退:

  • "On Completion" 依存を使用して、1 つのタスクが失敗してもチェーンを継続
  • 重要なタスクには代替スケジュールを作成
  • エスカレーション手順を文書化

3. リソース管理​

同時実行タスク:

  • 多すぎるタスクを同時にスケジュールしない
  • システムリソース（CPU、メモリ、ネットワーク）を考慮
  • 類似タスクをずらして競合を防止

プロキシのローテーション:

  • 同時にスケジュールされたタスクに十分なプロキシを確保
  • プロキシプール全体に負荷を分散
  • スケジュールされたタスクのプロキシの健全性を監視

4. テスト​

スケジュールのテスト:

  • スケジュール前に "Run Now" でテスト
  • まず 1 回限りの実行で確認
  • ログで問題をチェック

Cron 式の検証:

  • "Next Run Time" プレビューで確認
  • まず短い間隔でテスト
  • タイムゾーン設定が正しいことを確認

依存関係のテスト:

  • 親タスクが正常に完了することを確認
  • 遅延設定をテスト
  • チェーンが期待通りに動作することを確認

5. ドキュメンテーション​

わかりやすいスケジュール名:

  • 目的と頻度を含む説明的な名前
  • タスクタイプとターゲットを含める
  • 例: 「毎日の Google 検索 - マーケティング代理店」

説明の活用:

  • 目的と期待される結果を文書化
  • 依存関係と関連性を記載
  • 特別な配慮事項を含める

タスクの目的のラベル付け:

  • 関連するスケジュールにタグやカテゴリを付ける
  • プロジェクトやキャンペーンでグループ化
  • 識別しやすくする

トラブルシューティング​

スケジュールが実行されない​

考えられる原因:

  • スケジュールが非アクティブまたは一時停止中
  • cron 式の設定ミス
  • スケジューラーサービスが実行されていない
  • システム時刻/タイムゾーンの問題

解決策:

  1. スケジュールのステータスが "Active" であることを確認
  2. cron 式の構文を確認
  3. スケジューラーサービスが実行中であることを確認
  4. システム時刻とタイムゾーン設定を確認
  5. 実行ログでエラーを確認

タスクが頻繁すぎる頻度で実行される​

考えられる原因:

  • cron 式が正しくない
  • 同一タスクに複数のスケジュールがある
  • cron 構文の誤解

解決策:

  1. cron 式を慎重に確認
  2. 重複スケジュールをチェック
  3. "Next Run Time" プレビューで確認
  4. まず長い間隔でテスト

依存関係が発火しない​

考えられる原因:

  • 親タスクが完了していない
  • 依存条件が間違っている
  • 遅延が長すぎるまたは短すぎる

解決策:

  1. 親タスクの実行履歴を確認
  2. 依存条件が期待する動作に一致しているか確認
  3. 遅延設定を調整
  4. 循環依存がないか確認

タスクに時間がかかりすぎる​

考えられる原因:

  • タスク設定が攻撃的すぎる
  • システムリソースが不足
  • ネットワークのボトルネック

解決策:

  1. タスクのスコープを縮小（ページ数、同時実行数など）
  2. オフピーク時間にスケジュール
  3. 実行間隔を延ばす
  4. システムパフォーマンスを確認

実行履歴が表示されない​

考えられる原因:

  • タスクが一度も実行されていない
  • 履歴がクリアされた
  • データベースの問題

解決策:

  1. 手動でタスクを実行してテスト
  2. タスクが実行されたことがあるか確認
  3. データベース接続を確認
  4. 必要に応じてスケジューラーサービスを再起動

高度なワークフロー​

ワークフロー 1: 自動リードジェネレーションパイプライン​

スケジュール 1: 毎日の検索

Cron: 0 9 * * 1-5 (平日 9:00)
Task: "marketing agencies [city]" の Google 検索

スケジュール 2: メール抽出（依存）

Trigger: スケジュール 1 の成功後
Delay: 0 分
Task: スケジュール 1 の結果からメールを抽出

スケジュール 3: メールキャンペーン（依存）

Trigger: スケジュール 2 の完了後
Delay: 60 分（抽出のための時間を確保）
Task: ウェルカムメールキャンペーンを送信

結果: 自動化された毎日のリードジェネレーションとアウトリーチ。

ワークフロー 2: 週次メンテナンス​

スケジュール 1: データベースクリーンアップ

Cron: 0 3 * * 0 (日曜 3:00)
Task: 古い完了済みタスクを削除

スケジュール 2: プロキシヘルスチェック（依存）

Trigger: スケジュール 1 の完了後
Task: すべてのプロキシをテストし、失敗したものを削除

スケジュール 3: レポート生成（依存）

Trigger: スケジュール 2 の完了後
Task: 週次利用レポートを生成

結果: 自動化された週次メンテナンスとレポート作成。

ワークフロー 3: マルチリージョン監視​

スケジュール 1: 米国東部監視

Cron: 0 */2 * * * (2 時間ごと)
Task: 米国東部のキーワードを検索

スケジュール 2: 米国西部監視

Cron: 0 */2 * * * (2 時間ごと、オフセット)
Task: 米国西部のキーワードを検索

スケジュール 3: ヨーロッパ監視

Cron: 0 */2 * * * (2 時間ごと、オフセット)
Task: ヨーロッパのキーワードを検索

結果: ずらしスケジュールによる継続的なグローバル監視。

他の機能との連携​

タスクスケジューラーは以下と連携します:

  • 検索エンジン: 定期的な検索をスケジュール
  • コンタクト抽出: 検索後に自動抽出
  • イエローページ: 定期的なディレクトリスクレイピング
  • 一括メール送信: 自動キャンペーン

スケジューリングについて理解できたら:

  • システム設定の構成
  • 完全なユーザーマニュアルの確認
